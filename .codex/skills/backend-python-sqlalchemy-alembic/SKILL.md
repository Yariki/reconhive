# SQLAlchemy 2.x and Alembic Best Practices

## SQLAlchemy 2.x Querying

- Use 2.x-style `select()` and `session.execute()` / `session.scalars()` in new code.
- Keep sessions scoped to a request/unit of work.
- Keep transaction boundaries explicit.
- Avoid global sessions.
- Use eager loading or projections to prevent N+1 queries.
- Prefer explicit columns/DTO projections for read APIs.
- Do not materialize large datasets before filtering or pagination.

## Async SQLAlchemy

- Use `AsyncSession` for async FastAPI services.
- Use `async with session.begin()` for transactional writes when appropriate.
- Do not share one session across concurrent tasks.
- Avoid lazy loading that performs implicit I/O unexpectedly.

## Model Rules

- Use database constraints for required invariants: unique, check, foreign key, not null.
- Use server defaults when the database owns the value.
- Use timezone-aware timestamps.
- Keep naming conventions for constraints to make Alembic autogenerate stable.
- Separate ORM models from API schemas.

## Alembic Rules

- Always inspect autogenerated migrations.
- Keep one logical schema change per migration when practical.
- Migrations must be deterministic: no environment-dependent logic unless guarded.
- Do not edit applied shared migrations; create a corrective migration.
- Handle data migrations in batches for large tables.
- Include downgrade only if the project policy requires it, but never fake an unsafe downgrade.
- Watch for multiple heads and merge intentionally.

## Safe Migration Pattern for Large Tables

1. Add nullable column or new table.
2. Backfill in batches outside a long exclusive lock.
3. Deploy app that writes both old and new fields if needed.
4. Validate data.
5. Add constraints/indexes concurrently where supported.
6. Remove old field in a later migration.

## Verification

```bash
uv run alembic current
uv run alembic heads
uv run alembic upgrade head
uv run pytest
```

For migration smoke tests:

```bash
uv run alembic downgrade -1
uv run alembic upgrade head
```

## Anti-Patterns

- Autogenerated migration committed without review.
- Business logic inside migration scripts.
- Session stored in module/global state.
- Implicit lazy loads in serialization.
- Query filters applied after `.all()` / `.scalars().all()`.
