# EF Core Best Practices

## DbContext Lifetime

- Use a short-lived `DbContext` per unit of work/request.
- Do not store `DbContext` in singletons.
- Use context pooling only when the app is safe for pooled context reuse.
- Keep transaction scope explicit for multi-step writes.

## Querying

- Prefer projections (`Select`) for read models and API responses.
- Use `AsNoTracking()` for read-only queries.
- Use `AsNoTrackingWithIdentityResolution()` when read-only identity resolution is needed.
- Eager-load related data intentionally with `Include`, filtered include, split queries, or projections.
- Review generated SQL for complex LINQ.
- Apply filters before materialization.
- Always paginate list endpoints.

## Writes

- Load only what is necessary to update.
- Use concurrency tokens for collaborative or critical updates.
- Use explicit unique constraints to enforce business uniqueness.
- Avoid relying only on application-level duplicate checks.

## Migrations

- Generate migrations from model changes, then review the generated operations.
- Keep schema and data migrations small.
- Avoid destructive changes without a staged migration plan.
- Do not edit migrations already applied to shared environments unless the team explicitly allows it.

## Performance Risks

- N+1 queries.
- Client-side evaluation caused by unsupported expressions.
- Large tracked graphs.
- Unbounded `ToListAsync()`.
- `Include` chains that explode row counts.
- Missing indexes for foreign keys and lookup predicates.

## Verification

```bash
dotnet ef migrations list
dotnet build
dotnet test
```

For hot queries, capture generated SQL and inspect the database execution plan.
